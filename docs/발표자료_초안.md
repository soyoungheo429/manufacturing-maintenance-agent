- **주제**: AWS Bedrock Agent를 활용해 제조 설비의 센서 데이터를 실시간 분석하고, 고장 원인 추론부터 정비 권고·부품 발주까지 자동화하는 지능형 유지보수 시스템

### 팀원 소개 및 역할
| 이름 | 역할 | 담당 |
|------|------|------|
| 허소영 | 클라우드 인프라 | AWS 리소스 구성 + Lambda (센서 수집 / 이상 필터 / 결과 저장) |
| 임수희 | AI/ML | 데이터 전처리 + Knowledge Base(RAG) + Bedrock Agent |

- 단순 임계값 경보를 넘어, **LLM이 고장 원인을 추론하고 근거 기반 정비 방안을 제시**
- 재고 확인부터 부품 발주까지 **사람 개입 없이 자율적으로 처리**하는 에이전트 구현

## 2. 문제 정의

### 현황 및 배경
- 제조 현장의 설비 고장은 **비계획 정지(unplanned downtime)**로 이어져 막대한 생산 손실 발생
- 설비 센서(온도·회전속도·토크·공구마모 등)는 쏟아지지만, 이를 **해석하고 판단하는 것은 여전히 사람 몫**

### 기존 서비스의 문제점
- **단순 임계값 경보**: "범위를 벗어났다"는 것만 알려줄 뿐, 왜 고장인지·무엇을 해야 하는지 알려주지 않음
- **정비 지식의 속인화**: 고장 원인 판단과 정비 방법이 숙련 기술자의 경험에 의존
- **대응 지연**: 이상 감지 → 원인 파악 → 부품 확인 → 발주까지 단계마다 사람이 개입해 시간 소요

### 문제 해결 필요성
- 센서 데이터를 **자동으로 해석**하고, 축적된 정비 매뉴얼·과거 이력을 근거로 **일관된 판단**을 내리는 시스템 필요
- 감지에서 조치까지의 **리드타임을 단축**하고, 정비 의사결정을 표준화

## 3. 서비스 소개

### 서비스 개요
설비 센서 데이터를 10분마다 수집하여 이상 여부를 1차 필터링하고, Bedrock Agent가 Knowledge Base(정비 매뉴얼·고장 이력)를 검색해 고장 유형과 위험도를 추론합니다. 그 결과를 바탕으로 정비 권고를 생성하고, 부품 재고를 확인해 부족 시 발주서까지 자동 생성합니다. 관리자는 웹 대시보드에서 전체 설비 상태와 AI 권고를 확인합니다.

### 핵심 기능
1. **실시간 센서 모니터링** — 5대 설비의 센서값을 주기적으로 수집
2. **이상 탐지 (전처리)** — 정상 범위 대비 이탈 센서 추출
3. **AI 고장 진단** — Bedrock Agent + Claude가 고장 유형(TWF/HDF/PWF/OSF/RNF) 추론 및 위험도(0~100) 산정
4. **RAG 기반 정비 권고** — 정비 매뉴얼·과거 이력을 검색해 근거 있는 조치 제안
5. **자율 부품 발주** — 재고 확인 후 부족 시 발주서 자동 생성
6. **웹 대시보드** — 설비 상태·위험도·권고·발주 현황 시각화

### 사용자 흐름
```
[자동] 10분마다 → 센서 수집 → 이상 필터 → AI 분석 → 저장 → (이상 시) 재고확인·발주
[수동] 대시보드 새로고침 → 동일 파이프라인 즉시 실행 (1분 쿨다운)
관리자 → 대시보드에서 설비 상태·AI 권고 확인 → 발주 승인/거절
```

### 차별점
- **"경보"가 아니라 "판단"**: 임계값 초과 알림을 넘어 원인·위험도·조치까지 제시
- **RAG 기반 근거 제시**: 매뉴얼·이력에서 검색한 실제 근거로 신뢰도 확보
- **완결형 자율 처리**: 감지 → 진단 → 권고 → 발주까지 에이전트가 자율 수행

## 4. 구현 내역 및 AWS 아키텍처

### 기술 스택
- **데이터셋**: AI4I 2020 Predictive Maintenance Dataset (UCI) — 10,000행 센서 데이터
- **AI 모델**: Anthropic Claude (AWS Bedrock)
- **AI 오케스트레이션**: AWS Bedrock Agent + Knowledge Base (RAG)
- **백엔드**: AWS Lambda (Python 3.12), boto3
- **데이터 저장**: Amazon DynamoDB, Amazon S3
- **스케줄링**: Amazon EventBridge
- **프론트엔드**: React + Vite + TailwindCSS

### 주요 기능 구현
- **센서 수집 Lambda**: S3의 센서 CSV에서 5대 설비 데이터 샘플링 (고장/정상 혼합으로 시연 다양성 확보)
- **이상 필터 Lambda**: 센서값을 정상 범위와 비교해 이탈 센서 추출 후 Bedrock Agent 호출
- **Bedrock Agent**: KB 검색 + Claude 추론으로 고장 유형·위험도·권고·필요부품 판단
- **Action Group Lambda**: 결과 저장(dynamo_save), 재고 확인(inventory_check), 발주 생성(create_order)
- **대시보드 Lambda**: DynamoDB 조회 결과를 프론트에 제공

### 서비스 아키텍처
```
EventBridge (10분) / 대시보드 새로고침
        │
        ▼
  sensor_read (S3 센서 수집)
        │  체이닝
        ▼
  anomaly_filter (이상 센서 추출 → Bedrock Agent 호출)
        │
        ▼
   Bedrock Agent
     ├─ Knowledge Base (RAG) 검색: 정비 매뉴얼 · 고장 이력
     ├─ Claude 추론: 고장 유형 / 위험도 / 권고 / 필요 부품
     └─ Action Group 호출
          ├─ dynamo_save     (분석 결과 저장)
          ├─ inventory_check (부품 재고 확인)
          └─ create_order    (재고 부족 시 발주)
        │
        ▼
  DynamoDB → dashboard_data → 웹 대시보드
```

### AWS 서비스 구성
| 서비스 | 용도 |
|--------|------|
| Amazon S3 | 센서 데이터 CSV, Knowledge Base 문서 저장 |
| AWS Lambda | 센서 수집·이상 필터·저장·재고·발주·조회 로직 |
| Amazon Bedrock (Agent + KB) | 고장 원인 추론 및 자율 도구 호출 오케스트레이션 |
| Amazon DynamoDB | 탐지 결과(detection-results), 발주(purchase-orders), 재고(parts-inventory) |
| Amazon EventBridge | 10분 주기 자동 실행 트리거 |
| Amazon API Gateway | 프론트엔드-Lambda 연동 |

## 6. 기대 효과 및 향후 계획

### 기대 효과
- **다운타임 감소**: 고장 예측·조기 대응으로 비계획 정지 최소화
- **정비 의사결정 표준화**: 숙련자 경험에 의존하던 판단을 AI가 일관되게 수행
- **리드타임 단축**: 감지→진단→발주 자동화로 대응 시간 대폭 감소
- **정비 지식 자산화**: 매뉴얼·이력을 KB로 축적해 조직 지식으로 활용

### 한계점
- AI4I 공개 데이터셋 기반 시뮬레이션 — 실제 설비 IoT 스트림 연동은 미구현
- 설비 대수·센서 종류가 데이터셋 범위로 제한됨
- Vision(이미지) 교차검증은 목업 수준으로 구현

### 고도화 계획
- 실제 IoT 센서(AWS IoT Core) 실시간 스트림 연동
- 시계열 예측 모델(SageMaker) 추가로 고장 시점 예측 정밀화
- Vision 모델 실연동으로 센서 + 이미지 멀티모달 확진
- 발주 시스템을 실제 ERP/구매 시스템과 연동

### 확장 가능성
- 설비 유형·산업군 확장 (CNC 외 다양한 제조 설비)
- 다수 공장·다수 라인으로 멀티 테넌트 확장
- 정비 이력 축적에 따른 KB 자가 강화 및 예측 정확도 향상
